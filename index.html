Architecture Overview

System at a Glance

Plutonal AI is an institutional-grade quantitative investment research platform. Numbers come only from computation engines — LLMs only synthesise and explain.

Prime Directive
Compute First, Converse Last
All numbers originate from deterministic engines. LLMs never generate financials.
Positioning
Bloomberg + GPT
Quantitative like Bloomberg Terminal, synthesised like ChatGPT. Neither alone.
Entry Point
POST /api/v1/query
api/routes/query.py — normalize session, cache lookup, invoke PipelineOrchestrator
Orchestrator
PipelineOrchestrator
core/pipeline_orchestrator.py — owns L0–L6 and L8–L10. CRA owns only L7.
Agent Brain
CRA v7.0
LangGraph StateGraph. 12-node flow. Task-driven dynamic agent selection.
Cache Stack
L0→L1→L2→L3→L4
In-memory → Redis → Pinecone → Semantic → PostgreSQL. 80%+ hit target.

Layer 7 · LangGraph v7.0

CRA 12-Node Workflow

The Chief Research Agent runs a fixed 12-node LangGraph StateGraph. Task planning is LLM-driven; agent selection is dynamic per query, not static. All selected agents run in parallel via asyncio.gather().

NODE 01
Check Cache
Check Redis and semantic cache before any processing
NODE 02
Parse Intent
Extract structured intent, entities, and constraints from query
NODE 03
Task Planning
LLM decomposes query → tasks → assigns specific agent per task with directive
NODE 04
Mandatory API Fetch
Fetches core market data required for all agent types
NODE 05
Parallel Data Fetch
Supplemental data + RAG retrieval (HyDE, multi-query, reranking) for MEDIUM/BIG queries
NODE 06
Parallel Agent Dispatch
All selected agents execute concurrently via asyncio.gather()
NODE 07
Validate Outputs
Schema and confidence validation of all agent responses
NODE 08
Aggregate Results
Merge agent findings, arbitrate contradictions, build unified insight
NODE 09
LHIS Planning
Optional multi-horizon planning (skipped for SMALL queries)
NODE 10
Generate Output
LLM synthesises final executive_summary, cra_synthesis, follow-up suggestions
NODE 11
Quality Validation
Self-validate numeric claims, confidence calibration
NODE 12
Compliance Validation
Language sanitization, prohibited phrase check, disclaimer attachment
Universal Agent Contract: Every agent receives (query_id, entity, context with analytical_directive) → returns {status, confidence, findings, signal, entity, key_metrics}. Shared infrastructure at agents/shared/indicators.py, agents/shared/llm_config.py, agents/shared/response_builder.py, agents/shared/prompts.py.

Layer 6 · No LLMs

8 Computation Engines

All numeric outputs originate here. Pure NumPy / Pandas / SciPy / StatsModels. LLMs never touch these numbers — they only explain results from these engines.

Anomaly Detection
scipy · scikit-learn (isolation forest)
Price/volume spikes, zero volume on trading days, OHLC violations, large price gaps, stale data, probable splits. Runs first before all other engines.
Technical Screening
pandas · numpy · ta-lib
Vectorised: RSI 14, MACD 12/26/9, Stochastic 14/3/3, SMA 50/200, EMA 9/21/50, ADX, Bollinger %B, OBV, A/D Line. <8s for 450 stocks × 15 indicators.
Fundamental Scoring
pandas · numpy
P/E, P/B, ROE, revenue growth, margin compression, earnings quality scoring, restatement detection, industry median benchmarking, outlier detection.
Composite Screening
pandas · numpy · scikit-learn
Multi-factor: Valuation×0.25 + Catalyst×0.25 + Sentiment×0.25 + Macro×0.25. Normalization, z-score weighting, signal agreement percentage.
Short Squeeze Detection
pandas · numpy · scipy
Squeeze_Score = SI×0.25 + Days-to-Cover×0.20 + Momentum×0.20 + Volume×0.15 + Options×0.10 + Borrow×0.07 + Technical×0.03.
Statistical Research
statsmodels · scipy · numpy
Granger causality (F-tests, p<0.05, lag 1–10), VAR with impulse response, rolling correlations (90-day), event studies (MacKinlay 1997).
Macro Sensitivity
statsmodels OLS regression
Rate beta computation, stock returns vs macro variable changes, sensitivity scores, regime-conditional regression, interest rate impact quantification.
Regime Classification
numpy · scipy
Classifies market periods: expansion / contraction / transition / high_volatility from price momentum and macro indicators. Used by L6.5 and L8.

Cross-Cutting Concerns

Supporting Systems

Six specialised subsystems that run alongside the main pipeline, providing cost tracking, knowledge retrieval, learning, optimisation, multi-horizon planning, and web access.

QPU
Query Processing Unit
Normalised cost unit: 1 QPU ≈ $0.0025 USD. Tracks tokens, API weight, model size, and vector retrieval per query.
  • calculator.py: QPU = f(tokens, API calls, model, retrieval)
  • balance_manager.py: Per-user, per-tier atomic deduction via PostgreSQL
  • middleware.py: Pre-call balance check; post-call QPU deduction
  • nightly_allocation_job.py: Daily resets (Free/Essentials/Professional/Institutional)
  • API: GET /api/qpu/balance/{user_id}, /transactions, /dashboard
RAG v2.0
Knowledge Retrieval
Dynamic retrieval from ~300-book Pinecone vector store. Adaptive router: FAST / STANDARD / DEEP by query complexity.
  • Multi-query: LLM generates multiple query variants
  • HyDE: Hypothetical document embedding (skipped in FAST)
  • Cross-encoder reranking: LLM scores 0–10, re-sort
  • Iterative confidence: DEEP mode re-retrieves if quality <7/10 (max 2 iterations)
  • Parent document expansion for neighbouring chunks
RL
Reinforcement Learning
Trains policies for agent selection/weighting and multi-horizon behaviour. FinancialAgentEnvironment: 7 agents, 70-dim state space.
  • RLTrainer: PPO/A2C/SAC/TD3 via Stable Baselines3
  • MLflow tracking for all training runs
  • FeedbackLoop: Register predictions → collect outcomes → compute rewards → replay buffer
  • API: GET /api/v1/performance/rl/stats
MoE
Meta-Optimisation Engine
Hyperparameter optimisation using Optuna TPE. Safe mode with rollback on regression.
  • Optimised params: agent weights (7 agents, normalised)
  • min/high confidence thresholds, ensemble_method
  • max_qpu_budget, min/max_agents, freshness_decay_days
  • API: POST /api/v1/moe/refresh-weights (admin)
LHIS
Long-Horizon Intelligence
Multi-horizon planning: Tactical (1-day) / Operational (1-month) / Strategic (1-year). Hierarchical consistency enforcement across all horizons.
  • Optional RL models per horizon
  • CRA integration: Optional LHIS planning node (skipped for SMALL queries)
  • API: GET /api/v1/performance/lhis/stats
Web Manager
Real-Time Search & WS
Web search via Brave (primary) / DuckDuckGo (fallback). SearchCache avoids duplicate queries within session. Real-time WebSocket via Redis Pub/Sub.
  • Used by check_agent (secondary LLM path) and CRA parallel_data_fetch
  • WebSocketManager: real-time streaming updates
  • API: WebSocket /api/v1/ws/{client_id}

Design Boundaries: Numbers ONLY from computation_engine + API data — LLMs only classify, synthesise, and explain. BLOCK at L1 → immediate return. REFRAME → pipeline continues with reframed_query. Mandatory clarification blocks pipeline at L4 guard. Per-ticker quality gate <0.4 → rejected before computation. Confidence propagated: data → L5 → L7 enriched_context → L9 decomposition → L10 metadata. Pipeline owns L0–L6 and L8–L10. CRA owns ONLY L7. asyncio.Lock(_stats_lock) protects PipelineOrchestrator._stats mutations.
